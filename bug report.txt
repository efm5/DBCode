BUGS вЂ” Will Cause Compile Errors or Runtime Crashes
1. HighlighterEngine.HighlightNow() вЂ” undefined field mCurrentLanguage
mCurrentLanguage doesn't exist in HighlighterEngine. Fix: replace with mLanguage.

8. ThemeColorProvider never populated вЂ” highlighting renders nothing
SyntaxHighlighterBase.ApplyHighlighting() calls ThemeColorProvider.Get(role) but nothing ever calls ThemeColorProvider.Set() when a theme is applied. All tokens get Color.Empty.    foreach (TokenKind token in Enum.GetValues<TokenKind>()) {
      ColorRole role = LanguageColorRegistry.Get(pLanguage)[token];

         pTheme.mHighlightColors[(int)pLanguage][(int)token]);
   }
}
9. Theme constructor вЂ” syntax token colors written to wrong array
mInterfaceColors[(int)ColorUsage.Unknown], mInterfaceColors[(int)ColorUsage.Keyword], etc. are set in the constructor, but the highlighter reads from mHighlightColors, not mInterfaceColors. These assignments are silently ignored. Fix: remove them from the constructor; the SetXxxThemeSyntaxColors() methods in ThemeBuiltIns correctly populate mHighlightColors.
10. ThemeColorProvider / LanguageColorRegistry architectural mismatch
LanguageColorRegistry maps TokenKind в†’ ColorRole per language, and ThemeColorProvider maps ColorRole в†’ Color. But mHighlightColors in Theme is indexed directly by [LanguageKind][TokenKind] в†’ Color, bypassing both intermediaries entirely. The ColorRole indirection layer adds complexity without benefit. Recommend: have SyntaxHighlighterBase read directly from mCurrentTheme.mHighlightColors[language][token], eliminating ThemeColorProvider and LanguageColorRegistry as runtime dependencies.
11. ThemeDefaults вЂ” built-in themes added to mThemes twice
ThemeBuiltIns.Create*Theme() defaults to pAdd = true, so each static property initializer in ThemeDefaults adds to mThemes immediately. Then ThemeRegistry.AddBuiltIns() adds them again. Deduplication catches it but it's wasteful and fragile. Fix: pass pAdd: false in all ThemeDefaults property initializers.
12. ThemeRegistry.Initialize() вЂ” redundant built-in population
AddBuiltIns() and the ThemeManager.LoadThemes() fallback both add the same 7 built-ins. Fix: ThemeManager.LoadThemes() should never return built-ins as a fallback вЂ” that's ThemeRegistry's job.

DEAD CODE вЂ” Safe to Delete
13. Dictionaries class вЂ” entire class is unreferenced. Duplicates LanguageColorRegistry exactly. Delete it.
14. DBCode.Syntax.CSharpTokenizer вЂ” the simple early version in the DBCode.Syntax namespace is superseded by the full reader-pipeline version in DBCode.Syntax.Tokenizing. Will cause ambiguity. Delete the simpler one.
15. SyntaxHighlighterEngine вЂ” confirmed dead (see bug 6). Delete it.

QUALITY / CORRECTNESS ISSUES
16. ThemeDiagnostics.ValidateColors() вЂ” false positives on unnamed black
ColorTranslator.FromHtml("#000000") produces unnamed black and would be flagged as a parse failure even though it's valid. The check needs refinement.
17. CSharpTokenizer (Tokenizing namespace) вЂ” missing @ verbatim string prefix handling in StringReader
StringReader handles @"..." and $"..." but not $@"..." or @$"..." (both valid C# 14). Minor gap.
18. NumberReader вЂ” no hex literal support
0xFF, 0b1010, and 0o77 (C# integer literals) are not recognized and will tokenize as Unknown. Worth adding.
19. LanguageRegistry вЂ” only CSharp has a real tokenizer
The other 13 languages have highlighters via HighlighterFactory but no tokenizers registered anywhere. They need either full tokenizers or at minimum a shared fallback tokenizer so HighlightNow() doesn't crash.
20. PreprocessorReader вЂ” scans backward past start of string
The backward scan scanIndex = index - 1 could theoretically be fine, but if pStartIndex is 0 and index is 0, it initializes scanIndex to -1 and the while loop condition scanIndex >= 0 exits immediately вЂ” which is correct. This one is actually safe, just worth a confirming note.
